# Orpheus — философия и принципы (проект)

## Миссия
Orpheus — приватная коммуникация “по умолчанию”: сообщения и звонки должны работать в реальности (плохая сеть, фоновые режимы), при этом содержание переписки и звонков не должно быть доступно серверу.

## Базовые принципы
- **Анонимная идентичность**: пользователь идентифицируется публичным ключом, без телефона/email.
- **Минимизация доверия к серверу**: сервер не должен быть источником “истины” по содержимому сообщений; клиент шифрует/дешифрует сам.
- **Privacy‑by‑design**: даже при уведомлениях и диагностике избегаем утечек содержимого (например, push без текста сообщения).
- **Надёжность как часть продукта**: реконнект, очереди, fallback по хостам — это не “позже”, а часть базового UX.
- **Контрактное мышление**: поведение фиксируется тестами; изменения без тестового подтверждения считаются риском регрессии.

## Не‑цели (важные ограничения)
- **Анонимность ≠ неуязвимость**: проект не гарантирует защиту от всех видов компрометации устройства (root/forensics).
- **E2E не означает “ничего не хранится на устройстве”**: локальная история сообщений — осознанный компромисс UX/приватности.
- **Duress mode — правдоподобное отрицание, а не стирание**: режим принуждения скрывает данные в UI, но не обязан уничтожать их на диске.

## Модель угроз (границы ответственности клиента)
Клиент проектируется так, чтобы:
- сервер не видел содержимого сообщений (E2E);
- пуш‑уведомления не раскрывали содержимое сообщения;
- при физическом доступе к разблокированному устройству был дополнительный слой защиты (PIN/lockout/duress/wipe).

Клиент **не может гарантировать** защиту, если:
- устройство скомпрометировано (root/malware);
- атакующий получает доступ к файловой системе/бэкапам и читает локальные данные;
- сетевой периметр/серверная валидация построены неправильно (это зона ответственности всей системы).
