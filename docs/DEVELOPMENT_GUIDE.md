# Orpheus Client — гайд для разработки

## Принципы
- **Минимальные изменения**: правим только то, что нужно для задачи.
- **Контракты поведения** фиксируются тестами (unit/widget).
- **Сервисы** не должны напрямую зависеть от UI.
- При изменениях публичного поведения/UX обновляются соответствующие разделы `docs/`.

## Структура проекта (кратко)
Подробно: `docs/ARCHITECTURE.md`.

- `lib/` — приложение (экраны, сервисы, модели, виджеты, тема)
- `test/` — unit/widget тесты
- `android/` — Android‑часть (Gradle, манифесты, ресурсы)
- `scripts/` — вспомогательные скрипты (hooks, автокоммит и т.п.)

## Процесс изменений
### Перед изменениями
- Понять затрагиваемые контракты (что должно оставаться верным).
- Определить, какие тесты подтвердят “сделано”.

### Во время изменений
- Обновлять/добавлять тесты вместе с кодом.
- Избегать “скрытых” изменений поведения.

### После изменений
- Прогнать тесты (см. `docs/testing/README.md`).
- Обновить артефакты процесса:
  - `CHANGELOG.md` (секция `Unreleased`)
  - `AI_WORKLOG.md`
  - `docs/` (если меняется “как работает/как пользоваться”)
